# Key Information Masterlist

This document is the working checklist of the most important information the Asingan Fire Station website needs before full implementation.

The goal is simple:

- finalize the institutional details first
- avoid guessing or inventing public information
- make sure the website is credible, complete, and blog-ready

## Status Legend

- `Documented` means already saved in project files
- `Partially documented` means we have some usable details but still need clarification
- `Needs content` means we still need real station input
- `Planned` means the section is part of the website strategy but not yet filled

## 1. Station Identity

What this section should answer:

- what the station is officially called
- where it is located
- what it exists to do
- what kind of public service role it plays

Required information:

- `official station name`
- `municipality / province`
- `short station overview`
- `short mission-style description`
- `public-service identity statement`

Current status:

- `Partially documented`

Notes:

- We already have strong working context about the station's role, but we still need the final station profile wording for the website.

## 2. Organization Structure

What this section should answer:

- who currently leads the station
- what major units or functional groups exist
- what each group handles

Required information:

- `current head of station`
- `major offices / teams / functional groups`
- `short description per group`
- `lead person per group` if public and confirmed
- `support members` only if public and confirmed

Current status:

- `Partially documented`

Notes:

- This is one of the highest-priority information gaps.
- A simple public-facing structure is enough for version one.
- The current head of station can already be carried from the saved fire marshal timeline.
- The section and position structure can now be derived from the user-provided organization charts.
- Personnel names from those charts should still be treated as outdated until updated assignments are provided.
- Current assignees should stay in the separate assignment layer, while historical role history should stay in the service-history layer.

## Working Rule

If a detail is not confirmed, we should not force it into the website just to make a section look complete.

This project is stronger when:

- the story is clean
- the details are believable
- the public information is accurate
- the design is built on real station identity
