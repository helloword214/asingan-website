# AGENTS

## Purpose

This is the primary repo-level instruction file for agents and collaborators working in this project.

Read this first before proposing edits, implementation direction, or content structure.

## Core Rule

Do not assume.

If a file, route, component, data source, section, requirement, or hierarchy has not been clearly shown in the repository or explicitly provided by the user, treat it as `unknown`.

## Required Workflow

1. Inspect the actual repository first.
2. Summarize what actually exists.
3. Confirm the exact target area.
4. Edit only after the target is clear.

## Non-Negotiable Rules

1. `No assuming`
   Never act like a page, component, feature, or file structure already exists unless it is visible in the repo or explicitly confirmed by the user.

2. `Wait for shown context`
   If the user says they will show the file structure, content, screenshots, or reference first, wait for that input before proposing implementation details.

3. `Do not invent missing structure`
   Do not fill gaps with invented pages, hierarchy, personnel, data, or frontend structure just to keep momentum.

4. `Use only grounded sources`
   Base replies only on:
   - visible repository files
   - saved project data
   - explicit user instructions

5. `Mark uncertainty clearly`
   If something is not verified, say `not found in the repo`, `not yet shown`, or `unclear`.

6. `Do not confuse planning with implementation`
   Planning docs, content guides, and data placeholders must not be treated as proof that the actual app structure already exists.

7. `Keep scope narrow`
   Focus only on the exact area the user named. Do not expand the task automatically.

8. `State assumptions explicitly`
   If a small assumption is truly necessary, state it clearly so it can be corrected quickly.

## Default Safety Rule

If speed and accuracy conflict, choose accuracy first.
